# สรุปแนวคิดสถาปัตยกรรมซอฟต์แวร์และการพัฒนาซอฟต์แวร์
(Software Architecture & Software Development Concepts Summary)

เอกสารฉบับนี้เป็นการรวบรวมและจัดระเบียบเนื้อหาเกี่ยวกับสถาปัตยกรรมซอฟต์แวร์ (Software Architecture) เครื่องมือในการพัฒนาระบบ และแนวคิดในการออกแบบระบบประมวลผลแบบกระจาย เพื่อใช้เป็นคู่มืออ้างอิงในการเรียนรู้และการทำงาน

---

# สรุป
## 1. รูปแบบสถาปัตยกรรมและการประมวลผลแบบกระจาย (Architectural Styles & Distributed Systems)

### Client-Server Architecture
* **ความเหมาะสม:** รูปแบบสถาปัตยกรรมแบบ **Client-Server** เหมาะสมอย่างยิ่งสำหรับการพัฒนาระบบประมวลผลแบบกระจาย (Distributed Computing System) เนื่องจากมีการแบ่งแยกหน้าที่ระหว่างผู้ขอใช้บริการ (Client) และผู้ให้บริการ (Server) อย่างชัดเจน ช่วยให้สามารถกระจายภาระงาน (Workload) และการประมวลผลไปยังเครื่องคอมพิวเตอร์ต่าง ๆ ในเครือข่ายได้อย่างมีประสิทธิภาพ

### Microservices
* **แนวคิด:** เป็นการออกแบบสถาปัตยกรรมที่แบ่งแอปพลิเคชันออกเป็นบริการย่อย ๆ (Services) ที่ทำงานเป็นอิสระต่อกัน (Loosely Coupled) แต่ละบริการจะดูแลฟังก์ชันการทำงานเฉพาะทาง (Single Responsibility) และสามารถพัฒนา, ทดสอบ, และติดตั้ง (Deploy) แยกกันได้ ช่วยเพิ่มความยืดหยุ่นและความสามารถในการขยายระบบ (Scalability)

### Event-Driven Architecture (สถาปัตยกรรมแบบขับเคลื่อนด้วยเหตุการณ์)
* **แนวคิด:** เป็นสถาปัตยกรรมที่ตอบสนองต่อการเกิดขึ้นของ "เหตุการณ์" (Event) ในระบบ โดยองค์ประกอบต่าง ๆ จะสื่อสารกันผ่าน Event สถาปัตยกรรมนี้เหมาะสำหรับระบบที่มีการทำงานแบบ asynchronous หรือระบบที่ต้องการความขยายตัวได้ง่าย
* **ตัวอย่างการใช้งาน:** การประมวลผลและจ่ายเงินเดือนในช่วงสิ้นปี (Salary at end of year) ซึ่งเป็นเหตุการณ์ที่เกิดขึ้นตามเงื่อนไขเวลาและกระตุ้นให้ระบบบัญชี ทรัพยากรบุคคล และระบบธนาคารทำงานร่วมกันเมื่อถึงเวลาที่กำหนด

## 2. การทำงานร่วมกันระหว่างระบบและการเชื่อมต่อ (Interoperability & Integration)

### คุณสมบัติด้าน Interoperability (การทำงานร่วมกันระหว่างระบบ)
หากต้องการให้ระบบซอฟต์แวร์มีคุณสมบัติด้าน Interoperability เพื่อทำงานร่วมกับระบบอื่นได้อย่างสมบูรณ์ จะต้องมีความสามารถในการเชื่อมต่อ 2 ระดับ ดังนี้:
1. **Syntactic Interoperability:** การที่ระบบใช้ภาษา โปรโตคอล หรือรูปแบบการสื่อสาร (Format) เดียวกัน ทำให้ระบบสามารถ "เปิดอ่านและสื่อสาร" ข้อมูลระหว่างกันได้ (พูดภาษาเดียวกัน/โครงสร้างเดียวกัน เช่น การใช้ JSON หรือ XML)
2. **Semantic Interoperability:** การที่ระบบเข้าใจ "ความหมาย" ของข้อมูลที่ต้องสื่อสารระหว่างกันได้อย่างถูกต้อง แม้ว่ารูปแบบภาษาหรือไวยากรณ์ (Syntax) ที่ใช้จะมีความแตกต่างกันก็ตาม (เข้าใจบริบทและความหมายของข้อมูลตรงกัน)

### เครื่องมือและสถาปัตยกรรมในการเชื่อมต่อระบบ (Technical Architecture)
ในการเชื่อมต่อระบบต่าง ๆ ทั้งภายในและภายนอกองค์กรเพื่อให้สามารถทำงานร่วมกันได้อย่างเป็นระบบ องค์กรจำเป็นต้องใช้เครื่องมือและแนวคิดในระดับสถาปัตยกรรมเทคนิค (Technical Architecture) ดังนี้:
* **Subdomain & Domain:** การแบ่งขอบเขตของธุรกิจออกเป็นส่วน ๆ เพื่อระบุหน้าที่และความรับผิดชอบของระบบ
* **Middleware (ซอฟต์แวร์ตัวกลาง):** เครื่องมือหรือซอฟต์แวร์ที่ทำหน้าที่เป็นตัวกลางในการเชื่อมต่อ แลกเปลี่ยนข้อมูล และประสานงานระหว่างแอปพลิเคชันหรือระบบที่แตกต่างกัน ช่วยลดความซับซ้อนในการเชื่อมต่อแบบ Point-to-Point

### การจัดอันดับวิธีการแลกเปลี่ยนข้อมูล (Information Transaction Methods)
การเปรียบเทียบวิธีการรับส่งข้อมูลที่ดีที่สุดไปจนถึงแย่ที่สุด โดยพิจารณาจาก **ความสามารถในการขยายระบบ (Scalability)** และ **ความปลอดภัย (Security)**:

| อันดับ | วิธีการแลกเปลี่ยนข้อมูล | คำอธิบายความเหมาะสม |
| :---: | :--- | :--- |
| **1 (ดีที่สุด)** | **Message Exchange through Message Broker** | มี Scalability สูงสุดเนื่องจากทำงานแบบ Asynchronous ช่วยลดการพึ่งพากันโดยตรง (Decoupling) และมีความปลอดภัยสูง |
| **2** | **API (เช่น REST API)** | เหมาะสำหรับการสื่อสารแบบ Real-time/Synchronous มีมาตรฐานความปลอดภัยที่ดี (เช่น OAuth2, TLS) แต่คู่สนทนาต้องพร้อมทำงานพร้อมกัน |
| **3 (แย่ที่สุด)** | **File Export and Import** | มี Scalability ต่ำ และมีความเสี่ยงด้านความปลอดภัยสูงในการจัดการไฟล์ (เช่น การส่งผ่าน FTP) มักใช้ในระบบ Batch แบบดั้งเดิม |

## 3. การออกแบบและการสร้างแบบจำลองระบบ (System Modeling & Diagramming)

### C4 Model
C4 Model เป็นแนวคิดการทำแผนภาพสถาปัตยกรรมซอฟต์แวร์โดยแบ่งมุมมองออกเป็น 4 ระดับ (Context, Containers, Components, Code)
* **Component Level (ระดับที่ 3):** เป็นระดับที่แสดงรายละเอียดส่วนประกอบภายใน Container (เช่น ภายใน Web Application หรือ Mobile Application) ว่าประกอบไปด้วยโมดูลหรือส่วนประกอบภายในใดบ้าง และแต่ละส่วนประกอบมีการปฏิสัมพันธ์กันอย่างไร

### UML (Unified Modeling Language) & Deployment Diagram
* **UML:** ภาษามาตรฐานที่ใช้ในการสร้างแบบจำลองภาพ (Visual Model) เพื่ออธิบาย วิเคราะห์ และออกแบบระบบซอฟต์แวร์
* **Deployment Diagram:** แผนภาพใน UML ที่ใช้สำหรับแสดงโครงสร้างสถาปัตยกรรมทางกายภาพของระบบ (Physical Deployment) รวมถึงฮาร์ดแวร์ (Nodes), เครือข่ายการเชื่อมต่อ, และการติดตั้งซอฟต์แวร์หรืออาร์ติแฟกต์ (Artifacts) ลงบนเครื่องเซิร์ฟเวอร์หรือ Container ต่าง ๆ

### BPMN (Business Process Model and Notation)
* **แนวคิด:** มาตรฐานการเขียนผังงานเพื่อจำลองและอธิบายลำดับขั้นตอนการทำงานในกระบวนการทางธุรกิจ (Business Process) เพื่อให้ทั้งนักวิเคราะห์ธุรกิจ (Business Analyst) และนักพัฒนาระบบ (Developer) เข้าใจกระบวนการทำงานตรงกัน

## 4. เครื่องมือและแนวคิดในการพัฒนาซอฟต์แวร์ (Development Frameworks & Tools)

### CRUD Operation
* **ย่อมาจาก:** **C**reate, **R**ead, **U**pdate, **D**elete ซึ่งเป็นฟังก์ชันพื้นฐาน 4 ประการที่ระบบจัดการฐานข้อมูลและแอปพลิเคชันทั่วไปจำเป็นต้องมีในการจัดการกับข้อมูล

### Java Persistence API (JPA) / Hibernate Annotation
* **`@Id`:** เป็น Annotation ที่ใช้ในการกำหนด Attribute (ฟิลด์) ของ Entity Class เพื่อระบุว่าฟิลด์นั้นทำหน้าที่เป็น **Primary Key** ของตาราง (Data Table) ในฐานข้อมูลเชิงสัมพันธ์ (Relational Database)

### Lombok
* **แนวคิด:** ไลบรารีของ Java ที่ช่วยลดโค้ดที่ซ้ำซาก (Boilerplate Code) โดยใช้ Annotation (เช่น `@Data`, `@Getter`, `@Setter`, `@NoArgsConstructor`) ในการสร้าง Method พื้นฐานอย่าง getter, setter, constructor, หรือ toString ให้โดยอัตโนมัติในตอน compile

### Maven (mvnw)
* **แนวคิด:** เครื่องมือสำหรับบริหารจัดการโครงการซอฟต์แวร์ (Project Management Tool) ยอดนิยมในสาย Java ใช้สำหรับจัดการ Dependencies, Build, และ Test ระบบ 
* **Maven Wrapper (`mvnw`):** สคริปต์ที่ช่วยให้ผู้พัฒนาสามารถรันคำสั่ง Maven ได้โดยไม่จำเป็นต้องติดตั้ง Maven ลงในเครื่องคอมพิวเตอร์โดยตรง ช่วยควบคุมเวอร์ชันของ Maven ที่ใช้ในทีมให้ตรงกัน

### Container
* **แนวคิด:** เทคโนโลยีการจำลองสภาพแวดล้อมเสมือนในระดับระบบปฏิบัติการ (OS-level Virtualization) ที่รวมเอาแอปพลิเคชันและ Dependencies ทั้งหมดที่จำเป็นต้องใช้เข้าไว้ด้วยกันในแพ็กเกจเดียว (เช่น Docker) ช่วยให้แอปพลิเคชันสามารถรันได้เหมือนกันในทุก ๆ สภาพแวดล้อม (Development, Staging, Production)
